# Workflow: QA Agent - Execute Dynamic Testing

## 1. Objective & Scope
- **Purpose:** Orchestrate sandbox environment deployment, execute automated functional/regression test suites via native IDE terminal commands using specific capability manifests, and enforce the quality barrier before human code review.
- **Scope:** Runs during Phase 4 (Validation) inside the active feature branch. Authorized to write output logs/defects to `.backlog/` and clean test runtime storage. Restricted from altering `/src/` or `/docs/`.

## 2. Triggering Conditions & Inputs
- **Trigger:** Successful completion of `/.agents/workflows/qa-generate-test-suite.md` and verification that Programmer Agent code assets are compiled in `/src/`.
- **Inputs:**
  - Fresh automated test code built in `/tests/`.
  - Environmental deployment, sandbox infrastructure manifests, and test orchestration configurations located strictly in `/infra/`.
  - Authorized system tool access paths managed securely via `/.agents/mcp/mcp-settings.json`.

## 3. Step-by-Step Execution Lifecycle

### Phase 1: Environment Orchestration
1. **Analyze Infra Manifests:** Scan the `/infra/` directory to discover the deployment scripts (e.g., `docker-compose.yml`, shell wrappers).
2. **Boot Sandbox (SUT):** Invoke native IDE terminal utilities to execute sandbox environment orchestration commands, spinning up the System Under Test (SUT).
3. **Isolate External Dependencies (DOC):** Verify that all Dependent On Components (DOC) like external APIs, payment pathways, or remote third-party systems are successfully mocked or redirected to local test stubs.

### Phase 2: Test Suite Execution & Collection
4. **Trigger Test Runner:** Invoke the project-specific technical testing capability provisioned for this role. Execute the tool via the native terminal interface following the rules inside: `[Path to Specific Testing Skill]`.
5. **Monitor Telemetry:** Stream output runtime logs, capture database state alterations, and intercept edge-case panic exceptions.
6. **Calculate Quality Coverage Metrics:** Extract code coverage reports generated by the runtime test execution runner. Parse and validate that the strict framework gates are satisfied:
   - **100% Code Coverage** for the isolated Domain layer (`/src/{container}/domain/`), proving that all core business rules and edge cases are fully verified by the active test suite.
   - **>90% Code Coverage** for external infrastructure and application adapters (`/src/{container}/infrastructure/`, `/src/{container}/application/`).

### Phase 3: Defect Resolution Loop (If Tests Fail or Coverage Barriers Are Missed)
7. **Generate Structural Defect Report:** If any test case crashes, returns assertions failures, or fails to meet the 100%/90% coverage boundaries, halt the primary delivery sequence. Construct an explicit markdown defect log inside the active `.backlog/` feature file detailing:
   - Failing file path, line numbers, and exact stack trace error.
   - Missing coverage vectors (unexecuted lines or branches) within the `/src/{container}/domain/` or adapter folders.
   - Expected behavior vs actual observed system state.
8. **Signal Backtrack:** Route execution states back to the Programmer Agent in the chat, update `/.agents/memory-bank/active-context.md` to a blocking state, and await bugfix remediation. Loop back to Step 4 once a new build is signaled.

### Phase 4: Quality Gate (Human Verification & Sign-Off)
9. **Present Operational Report:** Once all test scenarios are green ("PASSED") and coverage metrics conform to constraints, compile the terminal output execution trace.
10. **Await Human Audit:** Pause execution. Present the clean test logs, SUT deployment summaries, and architectural verification markers to the human supervisor for formal verification sign-off.

### Phase 5: Environment Teardown & Handoff
11. **Clean Workspace Sandbox:** Execute teardown and container termination wrappers via the native terminal to decommission the local SUT and DOC stubs safely.
12. **Authorize Transition:** Log validation clearance inside `/.agents/memory-bank/decision-log.md` and append the final clearance path signature to the active task tracking file, releasing the feature branch for the next verification flow.
